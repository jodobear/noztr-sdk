# noztr-sdk 0.1.0-rc.2

Date: 2026-03-26
Release type: rc

## Summary

`0.1.0-rc.2` is the current public release candidate for `noztr-sdk`.

It keeps the same broad whole-library scope established by `0.1.0-rc.1`, but updates the public
baseline after the deliberate post-`rc.1` cleanup wave, expanded migration coverage, and
post-wave cross-lane validation. This release should be read as the current evaluated repo state
for downstream adoption, not as a narrow feature drop.

## What This Library Is

`noztr-sdk` is the higher-level Zig SDK layer in the `noztr` ecosystem.

It owns reusable non-kernel relay, workflow, runtime, store, and transport composition above
`noztr-core`, while keeping product-specific policy and hidden runtime ownership out of the SDK.

It is intended for real clients, relays, signers, bots, services, and CLI tooling.

## Coverage Snapshot

This RC continues to expose a broad grouped public floor across the current implemented line,
including:

- relay runtime, auth, query, exchange, replay, publish, response, workspace, and session
  composition
- local-state, archive, checkpoint, remembered-workspace, and embedded durable store support
- remote signer workflow plus signer-tooling client composition
- mailbox `NIP-17` DM, legacy `NIP-04` DM, mixed-DM app-facing composition, and signer-backed
  mailbox authoring
- `NIP-03`, `NIP-05`, and `NIP-39` proof and identity flows
- `NIP-29` groups
- social profile, note, long-form, reactions, lists, comment/reply, highlight, contact-graph, and
  starter-only WoT routes
- `NIP-57` zap workflow and request-bound receipt validation
- `NIP-86` relay-management client composition
- shared HTTP and signed-post transport seams where reusable beyond one consumer

## What Changed In This RC

This RC is primarily a whole-library consistency and usability release.

Main changes since `0.1.0-rc.1`:

- proof, identity, `NIP-05`, signer, mixed-DM, and `group_fleet` route-local naming was cleaned
  up further
- grouped proof/identity planning routes are now more canonical and less dependent on old flat
  workflow-planning walls
- migration coverage expanded for the newer pre-stable cleanup families
- the post-`rc.1` cleanup wave was validated across product-surface, Zig-native,
  docs/discoverability, downstream-foundation, LLM/human usability, and guardrail audit lanes

This RC does not redefine the library’s scope. It makes the current scope more consistent and
easier to consume.

## Tradeoffs And Limitations

- this line remains pre-`1.0`; compatibility should still be read conservatively
- `noztr-sdk` still prefers explicit caller-owned control over hidden runtime ownership
- not every NIP or future ecosystem capability is implemented
- downstream evaluation and real generic pressure still matter more than speculative new surface
  growth

## Breaking-Change Guidance

`0.1.0-rc.2` includes additional pre-stable cleanup relative to `0.1.0-rc.1`.

Use the canonical migration route first:
- [docs/reference/migration-guide.md](../reference/migration-guide.md)

That guide now routes the current breaking cleanup families in one place instead of forcing
downstreams to reconstruct the rename history from scattered notes.

## Compatibility Notes

- Zig toolchain floor for this RC line remains `0.15.2`
- `noztr-core` is expected at `v0.1.0-rc.6`
- `noztr-core` remains the deterministic protocol-kernel floor
- `noztr-sdk` remains the higher-level reusable composition layer above it

## Docs And Examples
