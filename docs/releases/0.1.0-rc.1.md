# noztr-sdk 0.1.0-rc.1

Date: 2026-03-25
Release type: rc

## Summary

`0.1.0-rc.1` is the first intentional public release candidate for `noztr-sdk`.

It establishes `noztr-sdk` as the higher-level Zig Nostr SDK above `noztr-core`, with a
documented public route, grouped namespace shape, compile-verified examples, migration guidance,
and a conservative pre-`1.0` compatibility posture. This RC is meant to be evaluated as the first
public versioned cut of the SDK as a whole, not as a single-feature release.

## What This Library Is

`noztr-sdk` is the higher-level Zig SDK layer in the `noztr` ecosystem.

It is designed to provide:

- reusable non-kernel relay, workflow, runtime, store, and transport composition
- caller-owned bounded control over network work, checkpoints, and runtime steps
- app-facing and SDK-facing surfaces that absorb broadly reusable Nostr heavy lifting above the
  kernel

It is not trying to be:

- a replacement for `noztr-core`
- a hidden websocket or daemon framework
- a product-specific app shell
- a place to hide product policy behind implicit runtime ownership

Those product and application concerns belong above the SDK.

## Coverage Snapshot

This RC exposes a broad grouped public floor across the current implemented line, including:

- relay runtime, auth, query, exchange, replay, publish, response, workspace, and session
  composition
- local-state, archive, checkpoint, remembered-workspace, and embedded durable store support
- remote signer workflow plus signer-tooling client composition
- mailbox `NIP-17` DM, legacy `NIP-04` DM, mixed-DM app-facing composition, and signer-backed
  mailbox authoring
- `NIP-03`, `NIP-05`, and `NIP-39` proof and identity flows
- `NIP-29` groups
- social profile, note, long-form, reactions, lists, comment/reply, highlight, contact-graph, and
  starter-only WoT routes
- `NIP-57` zap workflow and request-bound receipt validation
- `NIP-86` relay-management client composition
- shared HTTP and signed-post transport seams where reusable beyond one consumer

## Public Highlights

- first public RC tag for the documented `noztr-sdk` surface
- grouped public namespace shape is now the canonical discovery route
- broad naming and surface-noise remediation was completed before the first RC
- the public migration route is now explicit through
  [docs/reference/migration-guide.md](../reference/migration-guide.md)
- relay-management now shares the generic `transport.nip98_post` seam rather than carrying a
  private signed-`POST` stack

## Tradeoffs And Limitations

- this line remains pre-`1.0`; compatibility should still be read conservatively
- `noztr-sdk` prefers explicit caller-owned control over hidden runtime ownership
- not every NIP or future ecosystem capability is implemented
- lasting Blossom protocol/product surface is intentionally still downstream rather than absorbed
  here by default

## Compatibility Notes

- Zig toolchain floor for this RC line is `0.15.2`
- `noztr-core` remains the deterministic protocol-kernel floor
- `noztr-sdk` and `noztr-core` are complementary layers, not competing ones
- use local `../noztr-core` `master` until the next core tag:
  - `v0.1.0-rc.5` shipped a bad `zap_build_coordinate_tag` error-mapping form that is fixed on
    `master`

## Docs And Examples

- public docs now route through `README.md`, `docs/INDEX.md`, and `docs/getting-started.md`
- the contract map is the canonical task-to-symbol route
- the examples catalog is route-first at the top and detailed below that

## Adoption Guidance

- adopt this line as an RC, not as a long-established stable compatibility baseline
- start with the public routing docs instead of private maintainer context

- use the migration guide if you previously consumed older pre-`1.0` names
