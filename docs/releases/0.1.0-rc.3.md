# noztr-sdk 0.1.0-rc.3

Date: 2026-03-26
Release type: rc

## Summary

`0.1.0-rc.3` is the current public release candidate for `noztr-sdk`.

This is a targeted downstream-consumability RC. It keeps the `0.1.0-rc.2` public SDK surface, but
fixes the published package manifest so remote Zig package consumers no longer fail on a sibling
`../noztr-core` dependency path.

## What Changed In This RC

Main change since `0.1.0-rc.2`:

- published `build.zig.zon` now pins `noztr-core` as a remote dependency instead of a sibling path

This release is intentionally narrow. It does not reopen the public cleanup/remediation wave or
change the main grouped route structure.

## What This Library Is

`noztr-sdk` is the higher-level Zig SDK layer in the `noztr` ecosystem.

It owns reusable non-kernel relay, workflow, runtime, store, and transport composition above
`noztr-core`, while keeping product-specific policy and hidden runtime ownership out of the SDK.

It is intended for real clients, relays, signers, bots, services, and CLI tooling.

## Coverage Snapshot

The grouped public floor remains broad, including:

- relay runtime, auth, query, exchange, replay, publish, response, workspace, and session
  composition
- local-state, archive, checkpoint, remembered-workspace, and embedded durable store support
- remote signer workflow plus signer-tooling client composition
- mailbox `NIP-17` DM, legacy `NIP-04` DM, mixed-DM app-facing composition, and signer-backed
  mailbox authoring
- `NIP-03`, `NIP-05`, and `NIP-39` proof and identity flows
- `NIP-29` groups
- social profile, note, long-form, reactions, lists, comment/reply, highlight, contact-graph, and
  starter-only WoT routes
- `NIP-57` zap workflow and request-bound receipt validation
- `NIP-86` relay-management client composition
- shared HTTP and signed-post transport seams where reusable beyond one consumer

## Tradeoffs And Limitations

- this line remains pre-`1.0`; compatibility should still be read conservatively
- `noztr-sdk` still prefers explicit caller-owned control over hidden runtime ownership
- not every NIP or future ecosystem capability is implemented
- downstream evaluation and real generic pressure still matter more than speculative new surface
  growth

## Breaking-Change Guidance

There is no intentional public namespace or workflow break relative to `0.1.0-rc.2`.

Use the canonical migration route when evaluating older pre-`1.0` surfaces:
- [docs/reference/migration-guide.md](../reference/migration-guide.md)

## Compatibility Notes

- Zig toolchain floor for this RC line remains `0.15.2`
- `noztr-core` is expected at `v0.1.0-rc.6`
- published remote consumption is now supported for `noztr-sdk` itself

## Adoption Guidance

- treat `0.1.0-rc.3` as the current remote-consumable downstream evaluation line
- start with the public routing docs rather than maintainer state
- prefer grouped client/workflow routes
- if you hit a real downstream blocker on this line, route that evidence back into a targeted RC
  follow-up rather than rebuilding generic Nostr substrate locally
